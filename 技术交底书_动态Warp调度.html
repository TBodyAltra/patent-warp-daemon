基于运行时动态调度的 Warp 分工方法及系统

一、发明名称

一种基于运行时动态调度的 Warp 分工方法及系统

英文：Method and System for Runtime Dynamic Warp Scheduling in GPU Kernels

二、技术领域

本发明涉及 GPU 编译器优化技术，具体涉及一种在编译阶段划分 Kernel 分区并在运行时动态调度 Warp 执行的方法，适用于国产 GPU（如海光 DCU）或支持 Warp 级编程的异构计算设备。

三、背景技术

3.1 现有技术概述

当前 GPU 编程中，Warp-Specialize（线程束特化）是一种常见的优化技术。其核心思想是将一个 Warp 组的线程划分为不同的角色：

  • Producer（生产者）：负责内存访问和数据加载
  • Consumer（消费者）：负责计算密集型操作

这种分工模式可以在一定程度上实现计算与内存访问的重叠，提高硬件利用率。

3.2 现有技术的缺点

现有的 Warp-Specialize 技术存在以下不足：

问题 描述
静态划分 在编译阶段静态决定 Producer 和 Consumer 的分工，无法适应运行时变化
负载不均衡 当 Producer 和 Consumer 工作量差异较大时，一方会空闲等待
资源利用率低 无法根据运行时硬件资源占用情况动态调整任务分配
缺乏灵活性 无法处理复杂的条件分支和动态数据依赖

图1：传统静态 Warp-Specialize 示意

如图1所示，传统方案中 Producer 和 Consumer 的角色在编译时确定，运行时无法调整。当 Producer 工作完成后，如果 Consumer 仍在工作，Producer 就会空闲；反之亦然。这种静态划分无法适应复杂的运行时环境。

四、发明内容

4.1 发明目的

本发明的目的是提供一种基于运行时动态调度的 Warp 分工方法及系统，以解决现有技术中静态划分导致负载不均衡、资源利用率低的问题。

4.2 技术方案概述

本发明的核心思想是：将 Kernel 在编译阶段按照使用的硬件单元划分为不同的分区（Partition），每个分区再按照数据维度切分为小的任务（Task）。在运行时，由一个特殊的 Warp 担任调度器（Daemon Warp），根据硬件资源占用情况和数据依赖关系，动态将任务分发给空闲的 Warp 执行。

🎯 核心创新点

  1. 编译时分区：将 Kernel 按硬件单元使用划分
  2. 编译时任务切分：按数据维度切分 Task，确定输入（地址、参数）
  3. 运行时 Daemon Warp：动态调度，响应硬件状态变化
  4. 共享内存状态管理：Task 输入存储在共享内存，Daemon 通过状态机调度

4.3 详细技术方案

4.3.1 编译阶段

步骤1：Kernel 分区划分

在编译阶段，分析 Kernel 代码，识别不同类型的硬件操作：

  • 内存访问操作（Load/Store）→ 内存分区
  • 计算密集型操作（MFMA、ALU）→ 计算分区
  • 混合操作 → 混合分区

图2：编译阶段 Kernel 分区划分

步骤2：Task 切分与输入预计算

每个分区按照数据维度进一步切分为多个 Task，并预计算每个 Task 需要的输入：

// 编译生成的 Task 结构体（伪代码） struct TaskDescriptor { uint32_t task_id; // Task 唯一标识 uint32_t partition_id; // 所属分区 uint32_t input_ptr; // 输入数据指针（共享内存偏移） uint32_t output_ptr; // 输出数据指针 uint32_t param_offset; // 参数在共享内存的位置 uint32_t depend_mask; // 依赖 Task 掩码 uint32_t ready_mask; // 前置依赖完成状态 void (*kernel_func)(void*); // Task 执行函数指针 }; // Task 依赖关系示例 // Task 0: 无依赖 → 最先执行 // Task 1: 依赖 Task 0 → Task 0 完成后才能执行 // Task 2: 依赖 Task 0, 1 → 两者都完成后才能执行 // ...

步骤3：生成调度代码

编译阶段生成两部分代码：

  1. Task 执行代码：每个 Task 的具体实现
  2. Daemon 调度代码：运行时调度逻辑

// Daemon Warp 调度主循环 __device__ void daemon_scheduler() { while (!all_tasks_completed()) { // 1. 监控硬件资源占用 HardwareStatus hw_status = monitor_hardware(); // 2. 查找空闲 Warp int idle_warp = find_idle_warp(); if (idle_warp == -1) { continue; // 没有空闲 Warp，等待 } // 3. 根据硬件状态选择分区 int partition = select_partition(hw_status); // 4. 在选中分区中找满足依赖的 Task Task* task = find_ready_task(partition); if (task == nullptr) { continue; // 没有就绪 Task } // 5. 分发任务给空闲 Warp dispatch_task(idle_warp, task); } } // 分区选择策略 __device__ int select_partition(HardwareStatus hw) { if (hw.memory_util < 0.5 && hw.compute_util > 0.8) { return PARTITION_MEMORY; // 计算忙，分配内存任务 } else if (hw.memory_util > 0.8 && hw.compute_util < 0.5) { return PARTITION_COMPUTE; // 内存忙，分配计算任务 } else { return PARTITION_HYBRID; // 均衡 } }

五、具体实施方式

5.1 硬件平台

本发明适用于支持 Warp 级并行和共享内存的 GPU 架构，包括但不限于：

  • 海光 DCU 系列
  • AMD CDNA 系列
  • NVIDIA Volta 及以后架构

5.2 编译器实现

编译器需要实现以下模块：

  1. Kernel 分析器：分析 IR，识别硬件操作类型
  2. 分区生成器：按硬件单元划分分区
  3. Task 切分器：按数据维度切分 Task
  4. 输入预计算器：预计算 Task 输入（地址、参数）
  5. 代码生成器：生成 Task 执行代码和 Daemon 代码

5.3 调度策略

Daemon Warp 可以采用多种调度策略：

策略 描述 适用场景
负载均衡策略 选择硬件利用率低的分区 通用场景
依赖优先策略 优先选择依赖已满足的 Task 数据依赖强的场景
贪心策略 选择最早可完成的 Task 延迟敏感场景

六、优点

  1. 高资源利用率：通过动态调度，始终保持硬件资源充分利用
  2. 负载均衡：自动适应不同阶段的负载变化
  3. 高灵活性：可处理复杂的条件分支和动态数据依赖
  4. 易于编程：编译器自动完成分区和 Task 切分，开发者无需手动管理
  5. 可扩展性强：可容易地添加新的调度策略
